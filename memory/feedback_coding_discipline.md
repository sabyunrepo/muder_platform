코드 작업은 4단계 규율을 거친다.

**Why:** LLM이 가장 자주 만드는 실수 — (1) 가정을 숨긴 채 구현 진입, (2) 요청 범위 밖 추상화·기능 추가, (3) 인접 코드까지 "개선" (refactor 침범), (4) 성공 기준 없이 작업 시작. 4원칙은 이 순서대로 사후 보정 비용을 줄인다.

**How to apply:**

### 1. Think Before Coding (구현 전 사고)
- 가정은 명시적으로 진술. 불확실하면 질문.
- 해석이 둘 이상이면 사용자에게 제시 — 조용히 하나 고르지 않는다.
- 더 단순한 접근이 있으면 말한다. 정당하면 푸시백.
- 모호하면 멈추고 무엇이 모호한지 이름 붙여서 묻는다.

### 2. Simplicity First (단순함 우선)
- 요청한 것 외 기능 추가 금지.
- 1회용 코드에 추상화 금지.
- 요청되지 않은 "유연성"·"설정성" 금지.
- 발생 불가 시나리오용 에러 처리 금지.
- 200줄 짠 게 50줄로 가능하면 다시 쓴다.
- 자가질문: "시니어가 이걸 보면 over-engineered라 할까?" → 예 → 단순화.

### 3. Surgical Changes (외과적 변경)
- 기존 코드 편집 시: 인접 코드·주석·포맷 "개선" 금지.
- 망가지지 않은 것 리팩터 금지. 다르게 짰을지언정 기존 스타일 유지.
- 무관한 dead code 발견 시 **언급만** — 삭제하지 않는다.
- **내가 만든 orphan만 정리**: 내 변경이 unused로 만든 import·변수·함수만 제거. 기존 dead code는 요청 없으면 그대로.
- **검증 테스트**: 변경된 모든 라인이 사용자 요청에 직접 추적되는가?

### 4. Goal-Driven Execution (목표 기반 실행)
- 작업을 검증 가능한 목표로 변환:
  - "validation 추가" → "invalid input 테스트 작성 → 통과시키기"
  - "버그 고쳐" → "재현 테스트 작성 → 통과시키기"
  - "X 리팩터" → "전·후 테스트 통과 보장"
- 다단계 작업은 짧은 plan 진술:
  ```
  1. [step] → verify: [check]
  2. [step] → verify: [check]
  ```
- 강한 성공 기준 = 자율 루프 가능. 약한 기준("작동시켜")은 매 단계 재확인 유발.

**관련 카논**:
- 시작/완료 루틴 → `feedback_work_routine.md`
- 글로벌 추상 원칙 (Evidence/SOLID/DRY/KISS/YAGNI) → `~/.claude/CLAUDE.md` 핵심 원칙
- 4-agent 사후 리뷰 → `feedback_4agent_review_before_admin_merge.md`

**트레이드오프**: 신중함 > 속도. 사소한 작업은 판단으로 생략 가능.
