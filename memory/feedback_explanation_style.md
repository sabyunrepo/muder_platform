사용자에게 보고할 때 다음 3개 섹션 구조를 사용한다 — **원인 → 결과 → 권장 선택**. 추가로 **비개발자도 알아들을 수 있는 언어**로 풀어 쓴다.

**Why:** 2026-04-28 사용자가 명시 요청. 기존 진단 보고가 (a) 원인-결과-결정의 분리가 약하고, (b) 전문 용어 위주여서 즉각 판단이 어려웠음. 사용자는 결정자 역할이라 "선택지를 명확히 보여달라" + "기술 용어는 풀어달라"가 핵심.

**How to apply:**

1. **3섹션 구조 유지**
   - 🔍 **원인** — 무엇이 잘못됐고 왜 그런지 (한 문장 또는 표 1행)
   - 💥 **결과** — 그 결과 어떤 증상이 나타나는지 (사용자가 체감하는 것)
   - ✅ **권장 선택** — 다음에 뭘 할지 + 추천 이유 (선택지 2~3개일 때 권장 1개 강조)

2. **비개발자 친화 어휘**
   - 가능하면 비유 사용 ("docker는 컴퓨터 안의 작은 컴퓨터", "host port는 아파트 호수 같은 것")
   - 전문 용어 첫 등장 시 1줄 풀이 첨부 (예: "EACCES = 파일을 지울 권한이 없다는 OS 에러")
   - 코드/명령어는 박스 안에 넣고 그 위에 "무엇을 시키는 명령인지" 한 줄 설명
   - 약어 (CI, PR, UID, CVE 등)는 처음 등장 시 풀어쓰기 ("CI = 자동 검사 시스템")

3. **언제 적용**
   - 진단 보고 (CI 실패, 버그 분석 등)
   - 의사결정 요청 (옵션 A/B/C 제시)
   - design section 제시 (architecture, scope 등)
   - 예외: 사용자가 명시적으로 "기술 디테일까지" 또는 "raw" 요청 시 일반 모드로

4. **언제 적용 안 함**
   - 단순 명령어 실행 결과 echo (그대로 출력)
   - 코드 diff 자체 (코드는 코드대로)
   - QMD 회상 표 (이미 표 자체가 정형)

5. **template**
   ```
   ### 🔍 원인
   <무엇이 / 왜>

   ### 💥 결과
   <어떤 증상이 / 누가 영향받는지>

   ### ✅ 권장
   - (A) <선택지 1> — <장단점 한 줄>
   - (B) <선택지 2> — <장단점 한 줄> ⭐ 권장
   - (C) <선택지 3> — <장단점 한 줄>
   <왜 추천인지 한 줄>
   ```
