Q3. 컨셉 분류 — 사이드바를 어떻게 나눌까?

백엔드에 21+ ConfigSchema 모듈이 있어. 사용자(게임 디자이너) 머릿속에 어떻게 묶이느냐가 사이드바 자체를 정의함.

A. 모듈 카테고리 식 (백엔드 폴더 그대로)
🔍 단서 분배 (5)
starting / round / conditional / timed / trade
🚶 탐색 (4)
room / floor / timed / location_clue
🎴 단서 상호작용 (1)
clue_interaction
🏠 범죄현장 (3)
evidence / location / combination
⚖️ 결정 (3)
voting / accusation / hidden_mission
📜 진행 (7)
script / event / hybrid / gm / consensus / skip / ending
💬 커뮤니케이션 (5)
🎵 미디어 (1)
🔧 핵심 (4)

A. 모듈 카테고리 식 (9 카테고리)

백엔드 폴더 구조 그대로. 각 카테고리 안에서 모듈 ON/OFF.

장점

  • 백엔드와 1:1, 변환 비용 0
  • 새 모듈 추가 시 분류 결정 명확

약점

  • "단서를 어디서 편집"이 4 카테고리에 흩어짐 (분배·상호작용·탐색·범죄현장)
  • entity-first 의도와 어긋남
B. Entity 우선 (사용자 발화에 가까움)
📦 Entity
🔍 단서 (Clue)
📍 장소 (Location)
👤 캐릭터 (Character)
🎬 페이즈 (Phase)
🎭 결말 (Ending)
🌐 글로벌 시스템
⚖️ 결정 (투표/고소)
💬 커뮤니케이션
🎵 미디어
21 모듈 → entity 폼에 capability 로 붙거나, 글로벌에서 직접 ON/OFF

B. Entity 우선 (5 entity + 3 글로벌)

"내가 만들 게임의 등장 인물·장소·단서가 뭐냐"부터 시작. 모듈은 entity 에 능력으로 붙거나 글로벌 설정.

장점

  • 사용자 작업 흐름과 일치 (단서·장소·캐릭터부터)
  • entity 편집 페이지 = 사이드바 항목 1:1

약점

  • 모듈을 어느 entity 에 매핑할지 추가 결정 (예: round_clue = 단서? 페이즈?)
  • 글로벌과 entity-bound 구분 시 회색지대 존재
C. 게임 디자인 흐름 (작업 단계)
1️⃣ 세계 정의
캐릭터 / 장소 / 단서 (entity)
2️⃣ 게임 흐름
페이즈 / 라운드 / 진행 룰 (script/event/...)
3️⃣ 탐색·발견 룰
room/floor/timed exploration / location_clue
4️⃣ 단서 분배 룰
starting/round/conditional/timed clue
5️⃣ 결정·결말
voting/accusation/ending
6️⃣ 커뮤니케이션·연출
chat/voice/audio

C. 게임 디자인 흐름 (6 단계)

"세계를 만들고 → 흐름을 짜고 → 탐색·분배 룰 → 결말" 같은 작업 순서를 사이드바로. 위에서 아래로 따라가면 게임 1개 완성.

장점

  • 초보자가 길 잃지 않음 (튜토리얼 효과)
  • 1️⃣→6️⃣ 순서 = 자연스러운 작업 흐름

약점

  • 고수는 자유 점프 원할 수 있음 (작업 순서 강요 비호감)
  • 1️⃣ "세계 정의" 안에 entity 3종 (단서/장소/캐릭터) 또 분기 필요

하나 골라도 되고, 끌리는 거 여러 개 + 터미널에 "B 인데 C 의 작업 흐름 힌트도 살리고 싶다" 같은 변형 의견 적어도 돼.
