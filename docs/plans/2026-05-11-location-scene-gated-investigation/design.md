# Location Tab Rework — Scene-gated Investigation Design

## 활성 설계 요약

- `맵`은 장소를 묶는 관리/배치 컨테이너다.
- `장소`는 모두 동등한 조사 단위다. 부모 장소, 하위 장소, 3단계 장소 구조는 만들지 않는다.
- 조사 구조는 `맵 -> 장소 -> 단서` 2단계로 고정한다.
- 리스트형은 장소에 배치된 단서를 목록으로 보여준다.
- 카드덱형은 장소에 단서를 추가하면 그 단서가 곧 덱 카드가 된다.
- 카드덱형 MVP에서는 별도 슬롯 추가, 슬롯 라벨, 슬롯 수 관리 UI를 만들지 않는다.

## 원인

기존 장소 탭은 지도, 장소, 단서, 과거 부모/자식 장소 개념이 섞여 제작자가 어느 항목을 조사 단위로 봐야 하는지 혼란을 만들었습니다. 사용자가 최종 확정한 방향은 하위 장소를 없애고 장소 자체를 조사 단위로 두는 것입니다.

## 결과(목표 상태)

에디터에서 장소 탭을 통해 다음이 가능해야 합니다.

- 장소별 출현 장면과 숨김 장면 설정
- 장소별 접근 불가 캐릭터 설정
- 리스트형/카드덱형 조사 방식에 맞는 단서 배치
- 카드덱형 단서의 설정 순서/랜덤 지급 규칙 저장
- 단서 전역 중복 배치 방지
- 기존 round 기반 공개 설정과 parent location 유산의 점진 호환

## 데이터 모델

현재 MVP는 기존 `theme_locations`와 `modules.location.config.discoveries`를 유지하면서 scene 기반 필드를 추가합니다.

### `theme_locations`

- `id`
- `theme_id`
- `map_id`
- `name`
- `description`
- `appearance_scene_id`
- `hide_scene_id`
- `restricted_characters`
- `parent_location_id`는 기존 데이터 호환용으로만 남기고 신규 UI에서는 사용하지 않음

### `modules.location.config`

- `investigationMode`: `list` 또는 `deck`
- `deckOrder`: `fixed` 또는 `random`
- `discoveries`: 장소에 배치된 단서 목록

### 제약

- 한 단서는 하나의 장소 discovery에만 배치할 수 있다.
- 카드덱형에서 discovery 1개는 덱 카드 1장으로 해석한다.
- 별도 slot 엔티티나 slot 관리 UI는 만들지 않는다.
- 기존 parent 기반 데이터가 있더라도 신규 에디터 UI는 평면 장소 목록으로 다룬다.

## 노출 규칙

- `appearance_scene_id`가 비어 있으면 처음부터 공개로 해석한다.
- `hide_scene_id`가 비어 있으면 마지막 장면까지 공개로 해석한다.
- 장소 재출현은 지원하지 않는다.
- 조건 미충족 장소는 플레이 UI에서 보이지 않아야 한다.

## 탐색 모드

### 리스트형

- 장소에 배치된 단서를 목록으로 표시한다.
- 제작자는 장소 단위로 어떤 단서가 조사 가능한지 확인한다.
- 단서별 조사 비용과 필요 단서를 설정할 수 있다.

### 카드덱형

- 장소에 배치된 단서가 그대로 덱 카드가 된다.
- `deckOrder=fixed`면 현재 순서대로 한 장씩 지급한다.
- `deckOrder=random`이면 남은 단서 중 하나를 무작위로 지급한다.
- 드래그 앤 드롭으로 fixed 순서를 조정할 수 있다.

## 접근 제약

- 장소별 접근 불가 캐릭터를 설정한다.
- 접근 불가 캐릭터가 장소를 누르면 고정 문구를 표시한다.
- 문구: `해당 캐릭터는 탐색이 불가능한 장소입니다`

## 단서 소모/동시 처리

- 단서 1개는 전역 1회 획득이다.
- 동시에 여러 요청이 오면 서버 처리 요청순으로 먼저 처리된 플레이어가 먼저 획득한다.
- 남은 단서가 없으면 단서가 없다는 안내를 반환한다.
- 이번 PR 묶음은 에디터 저장 계약까지이며, 실제 런타임 소모 엔진은 후속 PR에서 처리한다.

## 마이그레이션 전략

- scene 기반 공개 컬럼을 점진 추가한다.
- 기존 round 기반 컬럼은 즉시 삭제하지 않는다.
- parent location 유산은 읽기 호환하되 신규 UI에서는 부모/하위 편집을 노출하지 않는다.
- 중복 단서 배치와 3단계 장소 구조는 저장 검증과 감사 쿼리로 탐지한다.

## 에디터 화면 제안

- 상단: 맵 선택 바와 장소 추가/삭제 액션
- 왼쪽: 현재 맵의 평면 장소 목록
- 중앙: 선택 장소의 단서 배치/덱 순서 캔버스
- 오른쪽: 선택 장소 설정 패널(공개 시점, 접근 제한, 조사 설정, 검수)

## 구현 우선순위

1. scene 기반 장소 공개 저장 계약과 마이그레이션
2. 장소 탭의 평면 장소 UI 고정
3. 단서 배치/해제와 전역 중복 방지
4. 카드덱형 단서 순서 드래그 앤 드롭
5. 수사 장면과 사용할 맵 연동
6. 런타임 단서 소모와 동시 처리 엔진
