# Deep Interview

모호한 요청을 바로 실행하지 말고, 실행 가능한 요구사항으로 정리한다.

핵심은 질문을 많이 하는 것이 아니라, 가장 큰 불확실성 하나를 골라 한 번에 하나씩 푸는 것이다. 바로 실행해도 안전한 기본값이 있고 사용자가 자율 진행을 허용한 상황이면 질문으로 멈추지 말고 권장 기본값으로 진행한다.

## 먼저 확인할 것

사용자에게 묻기 전에 직접 확인 가능한 맥락을 먼저 본다.

- 현재 repo의 `AGENTS.md`
- 관련 issue, PR, branch, plan 문서
- repo `memory/`와 프로젝트 규칙
- 코드베이스에서 확인 가능한 기존 패턴

코드나 문서로 답할 수 있는 질문은 사용자에게 묻지 않는다.

## 질문 축

아래 중 가장 중요한 불확실성 하나만 고른다.

- 목표: 무엇을 달성하려는가
- 범위: 포함할 것과 제외할 것은 무엇인가
- 제약: 지켜야 할 기술, 일정, 운영, 보안 조건은 무엇인가
- 완료 기준: 어떤 상태면 끝났다고 볼 수 있는가
- 영향 범위: 기존 기능, 사용자, PR, issue에 어떤 영향을 주는가

## 진행 방식

질문은 한 번에 하나만 한다.

OMX 환경에서 deep-interview 질문 라운드를 진행할 수 있으면 `omx question`을 사용한다. 그렇지 않은 환경에서는 아래 형식으로 사용자에게 직접 질문한다.

```md
현재 이해: {요청을 한 문장으로 요약}
막힌 결정: {가장 중요한 불확실성}
추천 답안: {있으면 제시}
질문: {한 가지 질문}
```

선택지가 도움이 되면 2-3개만 제시하고, 항상 자유 입력을 허용한다.

답변을 받으면 결정된 내용을 짧게 갱신한다. 아직 실행을 막는 중요한 불확실성이 남았을 때만 다음 질문을 한다.

## MMP 라우팅

요구사항이 정리되면 다음 작업 성격에 맞는 스킬로 넘긴다.

- GitHub issue 작성, 재작성, 병렬 작업 설계: `mmp-issue-planning`
- PR 생성, 리뷰 대응, merge 흐름: `mmp-pr-lifecycle`

## 종료 기준

다음이 정리되면 인터뷰를 멈춘다.

- 목표
- 포함 범위
- 제외 범위
- 제약
- 완료 판단 기준
- 남은 열린 질문
- 다음 권장 실행

마지막에는 전체 대화록이 아니라 아래 형식의 실행 브리프만 남긴다.

```md
## Execution Brief

목표:
범위:
제외:
제약:
완료 기준:
열린 질문:
다음 권장 실행:
```

변경, PR 생성, 배포, 삭제처럼 돌이키기 어려운 작업이 남아 있으면 실행 전 확인한다. 그 외에는 프로젝트 `AGENTS.md`와 사용자 지시에 따라 다음 단계로 진행한다.

## Avoid

- 구체적인 구현 요청, 작은 수정, 명백한 Git/PR 작업을 불필요한 인터뷰로 막지 않는다.
- 코드베이스에서 확인할 수 있는 내용을 사용자에게 다시 묻지 않는다.
- 한 번에 여러 질문을 던지지 않는다.
- 인터뷰 결과를 전체 대화록 형태로 길게 반복하지 않는다.
