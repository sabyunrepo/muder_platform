# 장소 관리 탭 UI/UX 재설계 실행 계획

## 0. 이번 설계의 고정 결정

- `맵`은 조사 계층이 아니라 장소를 묶는 관리/배치 컨테이너다.
- 장소는 모두 동등한 조사 단위다. 신규 UI에서는 장소 간 부모-자식 관계를 만들지 않는다.
- 조사 구조는 `맵 -> 장소 -> 단서`로 고정한다.
- `리스트형`은 장소에 배치된 단서를 목록으로 확인하고 획득하는 방식이다.
- `카드덱형`은 장소에 단서를 추가하면 그 단서가 자동으로 덱 카드가 되는 방식이다.
- 카드덱형 MVP에서는 별도 `슬롯 추가/슬롯 관리` UI를 만들지 않는다.
- 단서는 전역 유니크다. 한 번 배치된 단서는 다른 장소/덱에 중복 배치할 수 없다.
- 이번 범위는 에디터 UI/UX와 저장 계약 정리다. 플레이 런타임의 실제 단서 소모, 동시 클릭 충돌, 재접속 복원은 후속 runtime engine 작업으로 분리한다.

## 1. 목표

제작자가 장소 탭에서 다음 질문에 바로 답할 수 있게 만든다.

- 이 장소는 어느 장면부터 보이고 언제 숨겨지는가?
- 이 장소는 어떤 맵에 속하는가?
- 이 장소는 모든 캐릭터가 조사할 수 있는가?
- 리스트형이면 이 장소에 어떤 단서가 목록으로 배치되는가?
- 카드덱형이면 이 장소에서 어떤 단서 카드가 순차/랜덤으로 지급되는가?
- 현재 설정에서 빠진 단서와 빠진 공개 시점은 무엇인가?

## 2. 근거 기반 현재 상태

### 이미 있는 계약

- [x] `theme_locations.parent_location_id`는 기존 데이터 호환용으로만 유지한다.
- [x] `theme_locations.appearance_scene_id`
- [x] `theme_locations.hide_scene_id`
- [x] `theme_locations.restricted_characters`
- [x] `modules.location.config.investigationMode`
- [x] `modules.location.config.deckOrder`
- [x] `modules.location.config.discoveries`

### 현재 확인된 결함

- [x] `parent_location_id` 기반 UI는 제작자 혼란을 만들기 때문에 신규 에디터에서 제거한다.
- [x] 지도 이미지 선택 UI는 장소관리의 핵심 작업이 아니라 제거한다.
- [x] 장소 트리 안에 맵을 중첩하면 3단계 구조처럼 보여 제거한다.
- [x] 카드덱형 문구가 `슬롯` 중심이면 제작자가 단서만 추가하면 된다는 모델과 충돌한다.
- [x] 장소 트리거 UI는 현재 사용하지 않으므로 장소 탭에서 제거한다.

```text
[장소 관리 헤더]

┌──────────────┬────────────────────┬────────────────────┐
│ 장소 목록     │ 단서 배치 캔버스      │ 설정 패널             │
│              │                    │                    │

│ 경찰서        │ 거실의 단서 목록       │ 접근 제한             │
│ 창고          │                    │ 조사 설정             │
│              │ 카드덱형:            │ 검수                 │
│              │ 거실 단서 카드        │                    │
│              │ 1. 영수증            │                    │
│              │ 2. 사진              │                    │
└──────────────┴────────────────────┴────────────────────┘
```

## 4. UI 규칙

### 맵/장소 바

- [x] `맵`은 장소 목록의 루트가 아니라 상단 선택 바에 둔다.
- [x] 맵은 추가/삭제/선택만 제공한다.
- [x] 지도 이미지 선택 UI는 제공하지 않는다.
- [x] 장소 목록은 선택된 맵의 장소만 평면 리스트로 표시한다.
- [x] 장소 추가는 선택된 맵에 새 장소를 만든다.
- [x] 장소 목록에서 부모 경로, 하위 조사 항목, 2단계 문구를 노출하지 않는다.

### 단서 배치 캔버스

- [x] 리스트형일 때 중앙 캔버스 제목은 `전체 단서 목록`과 `조사 설정` 중심으로 둔다.
- [x] 리스트형은 장소에 배치된 단서를 선택하면 해당 단서 설정만 보여준다.
- [x] 카드덱형일 때 중앙 캔버스 제목은 `장소 덱 단서`로 표시한다.
- [x] 카드덱형은 `이 장소에 배치된 단서가 플레이 중 한 장씩 지급됩니다`를 설명한다.
- [x] 카드덱형은 별도 슬롯 추가 버튼을 만들지 않는다.
- [x] 카드덱형에서 `+ 단서 추가`로 단서를 추가하면 즉시 덱 카드가 된다.
- [x] `deckOrder=fixed`면 단서 순서를 번호로 표시하고 드래그 앤 드롭으로 조정한다.
- [x] `deckOrder=random`이면 단서를 `랜덤 후보` 목록으로 표시한다.

### 설정 패널

- [x] 선택한 장소의 설정만 보여준다.
- [x] 섹션 순서는 `기본 정보 -> 공개 시점 -> 접근 제한 -> 단서 배치 -> 검수`로 둔다.
- [x] 내부 필드명, raw JSON, config key는 노출하지 않는다.
- [x] `출현 장면`이 비어 있으면 `처음부터 공개`로 설명한다.
- [x] `숨김 장면`이 비어 있으면 `마지막 장면까지 공개`로 설명한다.
- [x] 접근 제한은 `조사 불가 캐릭터` 문구를 중심으로 표현한다.
- [x] 무료 조사는 체크박스로 제공하고, 체크 시 조사권 소비량 입력을 숨긴다.
- [x] 검수 항목은 클릭 시 관련 섹션으로 이동하거나 해당 섹션을 펼친다.

## 5. 데이터/검증 규칙

### 맵 -> 장소 -> 단서 구조

- [x] 신규 UI에서는 부모/하위 장소를 생성하지 않는다.
- [x] 서버는 legacy 3단계 장소 구조가 들어와도 신규 저장에서 다시 확대하지 않는다.
- [x] 장소는 선택된 맵에 속한다.
- [x] 단서는 하나의 장소 discovery에만 배치된다.
- [x] 단서 해제 시 해당 장소 discovery와 조사 비용 설정이 함께 제거된다.

### 리스트형 저장 의미

```ts
{
  locationId: 'living-room',
  clueId: 'receipt',
  order: 0,
  cost: { free: false, tokenCost: 1 }
}
```

- [x] `locationId`는 조사 가능한 장소다.
- [x] `clueId`는 전역에서 한 번만 배치된다.
- [x] 무료 조사 기본값은 해제이며, 무료 체크 시 조사권 소비량은 저장하지 않는다.

### 카드덱형 저장 의미

```ts
{
  locationId: 'living-room',
  clueId: 'receipt',
  order: 0
}
```

- [x] discovery 한 개가 덱 카드 한 장이다.
- [x] `slotId`, `label`은 MVP UI에 노출하지 않는다.
- [x] 순서 조정은 discovery order 변경으로 저장한다.
