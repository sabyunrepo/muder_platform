# 코딩 작업 완료 보고 — 6섹션 형식 강제

코드 변경(파일 신규/수정/삭제)이 포함된 task를 완료했을 때, atomic
commit 직후 사용자에게 결과를 보고하려면 아래 6섹션을 모두 채울 것.

발동 트리거: `TaskUpdate status=completed` + `git commit` 동시에 일어난
시점. 메타 작업(wrap-up·retro·doc 동기화·plan 작성)은 발동 X — 그건
별도 카논(`wrap-up-mmp`, `compound-wrap`)이 master.

## 발동 배경

2026-05-01 PR-9 Task 3.1 완료 직후 사용자 호소: "내가 말하면 너가 하는데
출력도 나오고 뭘 한 건지 모르겠어, 지금 블랙박스랑 같아." 그 자리에서
6섹션 형식으로 다시 풀어 설명했더니 이해 회복 — 이 패턴을 카논화.

원인 추정:
- 4섹션 보고(발견/수행/판단/미해결)는 *기술적으로* 정확하나 사용자가
  현재 위치(PR 안 어디, 다음 단계 무엇)를 잡지 못함
- jargon(interface, sqlc, testcontainers, pgtype) 이 그대로 노출
- "사용자 입장에서 무엇이 바뀌었나"가 빠져 있어 '아직 아무 변화 없음'을
  솔직히 말하지 못함 → 사용자가 작동하는 줄 오해
- 코드가 *재사용 가능한 부품*인지 *동작하는 기능*인지 구분 안 됨

## 6섹션 형식

### 1. 큰 그림 — 지금 만드는 기능이 무엇이고 왜 필요한가

현재 PR(또는 feature) 전체가 무엇을 만드는지 한 문단. 사용자 가치를
한 줄로. "왜" 를 빠뜨리지 말 것 (이전 결함, 표준 권고, 비즈니스 요구).

### 2. 이번 Task 가 PR 안에서 어디 위치

PR을 Task로 쪼갰다면 표 형식으로 진척 표시. 완료/이번/남은 Task를
명확히. 이번 Task가 "재료 준비"인지 "최종 와이어링"인지 한 줄.

### 3. 만든 파일 각각 (비유 사용)

신규/수정 파일별로 한 섹션. 각 섹션에:
- **비유 한 줄** — 현실 사물에 빗대어 (출입증·인터폰·시험대 등)
- 무엇을 하는가 (기술 설명 최소)
- 왜 만들었나 (이 파일이 없으면 어떤 일이 안 되는가)
- 호출되는 곳 (현재 누가 쓰는지, 미구현이면 "다음 Task X가 쓸 예정"
  명시)

### 4. 사용자 입장에서 무엇이 바뀌었나

**솔직히 답할 것**. 아직 사용자에게 보이는 변화가 없다면 "사용자
입장에선 아직 아무것도 바뀐 게 없습니다"라고 그대로 적을 것. 무엇이
*기술적으로* 추가됐고 *기능적으로* 무엇이 작동하지 않는지 구분해서.

### 5. 다음 Task 가 이 부품을 어떻게 쓸지 (미리보기)

Task X.Y 가 어떤 호출로 이번 산출물을 사용할지 한두 줄씩. 프리뷰가
구체적일수록 사용자가 흐름을 잡는다. "다음 세션 부품 → 요리" 식 비유 OK.

### 6. 커밋 위치 / 머지 상태

- 어느 branch / worktree 의 어떤 commit hash인가
- main에 머지됐나 (대부분 X — PR 생성 전)
- PR 생성·push 여부

## 톤·길이 가이드

- 분량: 6섹션 합쳐 본문 60~120 줄 권장 (코드 블록 제외)
- 톤: 사용자가 비개발자 친화 어휘를 선호한다는 `feedback_explanation_style.md`
  를 그대로 따른다 — 단 그건 *단일 사실 설명* master, 본 카논은 *작업
  완료 보고* master. 충돌 시 본 카논의 6섹션 구조를 우선.
- jargon 사용 시 비유나 한 줄 풀이를 짝지어라
- 이모지 금지 (사용자 글로벌 정책)

## 안티 패턴

- ❌ 4섹션(발견/수행/판단/미해결)만 출력 — 본 카논 섹션 1·2·4·5 누락
- ❌ "구현 완료" 만 적고 무엇이 작동하는지 안 적음
- ❌ 비유 없이 jargon 나열 (interface, struct, generic 등)
- ❌ "사용자 입장에선 아무것도 안 바뀜" 을 회피해 *부품 준비*를 *기능
  완성*인 것처럼 포장 → 다음 세션 사용자가 "왜 안 되지?" 혼란
- ❌ Task가 PR 안에서 어디 위치한지 누락 → 사용자가 진척 못 잡음
